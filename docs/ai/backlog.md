# Беклог отложенных оптимизаций (Технический долг)

Этот файл содержит список технических задач, оптимизаций и архитектурных улучшений, которые были обнаружены в ходе аудитов экранов, но сознательно отложены для обеспечения безопасности и пошаговости изменений.

---

## Главный экран (`home_screen.dart` / `Home`)

Задачи по мотивам аудита от 24 июня 2026 года:

### 1. 🟡 Реактивные соединения (Reactive Joins) в слое UI
* **Проблема**: `HomeTransactionListItem` (ранее `_TransactionListItem`) отслеживает (через `watch`) несколько провайдеров одновременно (транзакцию, аккаунт, категорию, теги, трансферный аккаунт). При отображении большого списка это порождает десятки/сотни активных подписок.
* **Что нужно сделать**: Перенести логику соединения таблиц (JOIN) на уровень базы данных/репозитория. Провайдер должен возвращать единую доменную сущность (например, `TransactionWithMetadata` или `FeedItem`), содержащую уже загруженные данные категории, аккаунтов и тегов.
* **Сложность**: Средняя (требует доработки слоя данных/UseCase).

### 2. 🔴 Отсутствие виртуализации в списке транзакций
* **Проблема**: Секция транзакций `HomeTransactionsSectionCard` рендерит элементы через обычный `Column`, помещенный в `SliverToBoxAdapter` главного скролла. Это отключает ленивый рендеринг (lazy loading) элементов во viewport-е.
* **Что нужно сделать**: Избавиться от двойной вложенности скролла. Интегрировать отображение транзакций напрямую в главный `CustomScrollView` через `SliverList` или `SliverGroup` для ленивой отрисовки.
* **Сложность**: Высокая (требует изменения верстки и структуры прокрутки экрана).

### 3. 🟡 Обновление `PageController` во время фазы Build (Antipattern)
* **Проблема**: В карусели счетов (`_AccountsListState`) пропорции viewport-а рассчитываются во время билда и обновляются через `WidgetsBinding.instance.addPostFrameCallback` с пересозданием `PageController` и `setState`, вызывая двойной проход рендеринга.
* **Что нужно сделать**: Инициализировать и обновлять параметры контроллера в методах жизненного цикла `didUpdateWidget` или рассчитывать стабильный `viewportFraction` заранее, без необходимости пересоздавать `PageController` реактивно на каждый билд.
* **Сложность**: Средняя.

### 4. 🟢 Оптимизация выделения памяти для DateFormat в `build`
* **Проблема**: В `_CreditCardContent.build` при каждом билде создается новый объект `DateFormat.yMd(strings.localeName)`.
* **Что нужно сделать**: Использовать кеширование (например, расширить существующий утилитный класс `TransactionTileFormatters`).
* **Сложность**: Низкая.
