# Стартовые auth entry-flow по runtime

Документ описывает, какой стартовый сценарий входа показывается пользователю в разных runtime-контурах приложения.

## Что изменилось

- `web` больше не показывает действие `Продолжить офлайн`.
- `firebaseDev` сначала требует активацию лицензионного ключа, и только после этого открывает вход или регистрацию.
- `firebaseProd` пока оставлен без изменений, включая текущий тестовый сценарий с автоподстановкой ключа на старте.
- `offline` flavor не использует экран входа как основной сценарий запуска.

## Зачем

- Исключить ложный offline-вход в `firebaseDev`, который создаёт `local-*` профиль и затем блокирует облачную синхронизацию.
- Сделать `web` однозначно облачным контуром без смешения с локальным режимом.
- Сохранить `firebaseProd` стабильным для текущего тестового процесса.

## Матрица поведения

### Web

- Доступны:
  - вход по email/паролю;
  - регистрация нового аккаунта.
- Недоступно:
  - `Продолжить офлайн`.

### FirebaseDev

- До активации ключа:
  - показывается экран ввода лицензионного ключа;
  - формы входа и регистрации скрыты.
- После активации ключа:
  - доступны вход по email/паролю и регистрация;
  - офлайн-кнопка не показывается.

### FirebaseProd

- Поведение экрана входа не меняется в рамках этой правки.
- Тестовый ключ может активироваться автоматически на старте, если это настроено в runtime.

### Offline flavor

- Основной сценарий — локальная работа без обязательного облачного входа.

## Как проверить

```bash
flutter run -d chrome
flutter run -d <device> --flavor dev
flutter run -d <device> --flavor prod -t lib/main_prod.dart
```

- На `web` убедиться, что на экране входа нет `Продолжить офлайн`.
- На `dev` убедиться, что первым показывается ввод ключа, а после `DEMO-CLOUD-KEY` становятся доступны вход и регистрация.
- На `prod` убедиться, что экран остался прежним.

## Breaking changes

- Для `firebaseDev` больше нельзя сначала уйти в offline flow с экрана входа и затем безопасно тестировать облако в той же установке.
