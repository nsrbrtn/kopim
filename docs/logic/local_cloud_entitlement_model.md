# Kopim: продуктовая и техническая модель локальной и онлайн-версии

## 1. Цель документа

Подготовить текущий рабочий продукт Kopim к чёткому разделению локальной и онлайн-версии, безопасному переходу пользователей из локального режима в облако, будущей подписочной модели и развитию ИИ-функций.

Важно: Kopim уже не MVP. Приложение опубликовано в Play Market, есть web-версия `kopim.site`, работает синхронизация и уже существуют флейворы:

* `offlineOnly` — отдельный автономный offline-only дистрибутив;
* `storeProdLocalFirst` — основной Android Play contour: local-first, но cloud-capable сборка с реальной базой данных;
* `dev` — версия для теста облака с базой данных, которую не жалко.

Задача не в том, чтобы спроектировать продукт с нуля, а в том, чтобы грамотно подготовить код, продуктовую логику и UI к будущей модели:

* локальная версия бесплатная;
* облако, web, ИИ и расширенная аналитика доступны по cloud-подписке;
* пользователь может начать локально без регистрации;
* позже пользователь может включить синхронизацию;
* при включении облака локальные данные не должны быть потеряны или случайно смешаны с облачными.

## 2. Продуктовая модель

### 2.1. Локальная версия

Локальная версия Kopim — это полноценное приложение без регистрации, синхронизации и ИИ.

В локальной версии:

* пользователь не регистрируется;
* данные хранятся только локально на устройстве;
* нет доступа к синхронизации;
* нет доступа к ИИ-ассистенту;
* нет доступа к web-синхронизации;
* работают все основные функции приложения.

В локальной версии должны работать:

* счета;
* категории;
* операции;
* переводы;
* бюджеты;
* регулярные платежи;
* аналитика;
* копилки;
* импорт/экспорт;
* виджеты;
* напоминания.

Локальная версия не должна ощущаться как демо или урезанная версия. Это полноценный бесплатный личный финансовый менеджер.

Основное сообщение для пользователя:

> Все данные хранятся только на этом устройстве.

---

### 2.2. Онлайн-версия

Онлайн-версия Kopim — это тот же продукт, но с облачными возможностями.

В онлайн-версии доступны:

* вход в аккаунт;
* синхронизация между устройствами;
* синхронизация с web-версией;
* доступ к `kopim.site`;
* ИИ-функции;
* расширенная аналитика;
* будущая cloud-подписка.

Главная ценность онлайн-версии:

* удобный вход с любого устройства;
* синхронизация между устройствами;
* доступ к web-версии;
* ИИ как часть cloud-подписки.

Онлайн-версия не должна быть отдельной логикой учёта. Это облачный слой поверх локального ядра.

---

## 3. Главный продуктовый принцип Kopim

Kopim — это личный финансовый помощник, который помогает удобно следить за финансами и даёт понятную аналитику.

Главный экран и логика приложения должны отвечать не только на вопрос:

> Что я записал?

Но и на вопросы:

> Сколько у меня денег?
> Где я близок к лимиту?
> На что стоит обратить внимание?
> Сколько я могу тратить дальше?
> Всё ли в порядке с моим бюджетом?

## 4. Архитектурная модель

Нужно разделить три независимые оси:

1. Build Flavor
2. Runtime Data Mode
3. Entitlement

Их нельзя смешивать в одну логику.

### 4.0. Fresh Upload после удалённого cloud-delete

Если remote metadata сообщает `cloudDataState == deleted`, повторное включение облака для локального пользователя идёт только через отдельный Fresh Upload flow:

1. remote metadata `deleted`;
2. transition `deleted -> freshUploadInProgress`;
3. controlled upload локального графа;
4. remote verification загруженного графа;
5. transition `freshUploadInProgress -> active`;
6. локальная idempotent finalization;
7. runtime `cloudEnabled`.

Remote metadata state, локальный marker finalization и runtime mode являются разными состояниями. `cloudDataState == active` означает, что удалённый граф принят как canonical, но не означает, что локальная ownership/outbox финализация уже завершена. Runtime `cloudEnabled` запрещён, пока локальный Fresh Upload finalization marker не сохранён.

Firestore Rules проверяют owner, active entitlement и допустимые metadata transitions. Они не доказывают успешную remote verification: verification выполняет клиентский orchestration service перед переводом metadata в `active`.

Локальная finalization должна быть idempotent и crash-safe. Если приложение упало после remote `active`, но до локального marker, следующий запуск обязан оставить обычный sync выключенным и повторить только локальную finalization.

---

### 4.1. Build Flavor

Build Flavor отвечает за технические возможности сборки.

#### `offlineOnly`

Отдельный автономный privacy/offline distribution.

Особенности:

* приложение работает локально;
* пользователь не регистрируется;
* синхронизация недоступна;
* ИИ недоступен;
* данные хранятся только на устройстве.

Важно: в будущем желательно прийти к одной публичной сборке, где из Play Market версии можно включить облако, если это не будет слишком сложно и не будет конфликтовать с правилами публикации и платежей.
В текущей продуктовой модели `offlineOnly` уже не считается основным Play contour и сохраняется как параллельный автономный вариант дистрибуции.

#### `storeProdLocalFirst`

Основной Android Play contour: local-first по UX, но cloud-capable по возможностям.

Особенности:

* Auth включён;
* cloud sync включён;
* AI включён;
* используется реальная Firebase-база;
* это текущий целевой публичный Play-контур;
* mobile runtime должен оставаться consumption-only: без in-app purchase CTA, promo/license UX и прямых payment flows.

#### `dev`

Версия для разработки и тестирования облака.

Особенности:

* Auth включён;
* cloud sync включён;
* AI включён;
* используется dev Firebase-база, которую не жалко;
* можно тестировать миграции, конфликты, синхронизацию, entitlement и подписки.

---

### 4.2. Runtime Data Mode

Runtime Data Mode отвечает за то, как сейчас работают данные пользователя.

Возможные внутренние состояния:

* `localOnly`;
* `cloudEnabled`;
* `cloudBlockedByLocalData`;
* `syncInProgress`;
* `syncError`;
* `syncConflict`;
* `entitlementRequired`;
* `entitlementExpired`.

Пользователю не нужно показывать все эти технические состояния.

В UI достаточно простых формулировок:

* `Локально`;
* `Синхронизация включена`;
* `Синхронизация выключена`;
* `Нужно действие`;
* `Ошибка синхронизации`.

---

### 4.3. Entitlement

Entitlement отвечает за то, какие платные или облачные функции доступны пользователю.

В будущем нужны состояния:

* `freeLocal`;
* `cloudTrial`;
* `cloudActive`;
* `cloudExpired`;
* `aiActive`;
* `unknown`.

ИИ не должен быть отдельной подпиской. Он входит в cloud-подписку.

Расширенная аналитика тоже относится к cloud-подписке.

Feature gates должны быть построены не вокруг вопроса “есть ли подписка?”, а вокруг вопроса “доступна ли функция?”.

Пример:

* `canUseCloudSync`;
* `canUseWebSync`;
* `canUseAiAssistant`;
* `canUseAdvancedAnalytics`.

## 5. Будущая подписочная модель

### 5.1. Бесплатно

В бесплатной локальной версии доступны:

* счета;
* категории;
* операции;
* переводы;
* бюджеты;
* регулярные платежи;
* копилки;
* базовая аналитика;
* импорт/экспорт;
* виджеты;
* напоминания.

Не нужно ограничивать количество операций, счетов или категорий.

---

### 5.2. Платно в cloud-подписке

В cloud-подписку входят:

* синхронизация между устройствами;
* доступ к web-версии `kopim.site`;
* ИИ-ассистент;
* расширенная аналитика;
* будущие облачные функции.

---

### 5.3. Пробный период

Нужно предусмотреть пробный период для cloud-функций.

Идея сценария:

1. В приложении пользователь видит состояние:

   > Синхронизация выключена

2. Рядом есть действие:

   > Включить

3. После нажатия пользователь попадает на страницу, где:

   * объясняется, что даёт онлайн-версия;
   * показываются преимущества cloud-подписки;
   * предлагается регистрация;
   * даётся подарочный пробный период;
   * пользователь получает код или активирует пробный доступ.

4. После активации entitlement приложение включает облачные функции.

Возможный текст:

> Включите синхронизацию, чтобы пользоваться Kopim на нескольких устройствах, открывать финансы в web-версии и получить доступ к ИИ-помощнику.

---

### 5.4. Если подписка закончилась

Если cloud-подписка закончилась:

На телефоне:

* приложение продолжает работать локально;
* новые операции сохраняются локально;
* пользователь не теряет доступ к своим данным;
* синхронизация ставится на паузу;
* ИИ становится недоступен;
* расширенная аналитика становится недоступна.

В web-версии:

* доступ становится read-only;
* пользователь может посмотреть данные;
* редактирование и создание новых данных недоступно;
* можно показать предложение продлить подписку.

Важно: окончание подписки не должно приводить к потере данных.

---

## 6. Переход из локального режима в облако

### 6.1. Базовый принцип

Если пользователь начал локально, а потом решил включить облако, приложение должно автоматически предложить перенести локальные данные в аккаунт.

Нельзя автоматически смешивать локальные и облачные данные без подтверждения пользователя.

Основной текст логики:

> У вас есть данные на устройстве. Чтобы не потерять их, выберите, что сделать.

---

### 6.2. Сценарий: локальные данные есть, облако пустое

Экран:

> Подключить синхронизацию

Текст:

> На этом устройстве уже есть данные. Их можно перенести в аккаунт, чтобы они были доступны на других устройствах.

Действия:

* `Перенести данные в аккаунт`;
* `Оставить только на устройстве`;
* `Подробнее`.

---

### 6.3. Сценарий: локальные данные есть, в облаке тоже есть данные

Экран:

> Найдены данные на устройстве и в аккаунте

Текст:

> На этом устройстве уже есть локальные данные, а в вашем аккаунте уже есть облачные данные. Чтобы ничего не потерять, выберите, как поступить.

Действия:

* `Объединить данные`;
* `Заменить данные на устройстве данными из аккаунта`;
* `Оставить данные только на устройстве`.

Автоматически объединять данные нельзя. Только после явного подтверждения.

---

### 6.4. Сценарий: cloud недоступен из-за flavor

Этот сценарий не нужен и должен быть убран из продуктовой модели.

В целевой логике cloud недоступен не потому, что “flavor запрещает пользователю функцию”, а потому что:

* пользователь не включил синхронизацию;
* пользователь не вошёл в аккаунт;
* пользователь не активировал trial или подписку;
* entitlement отсутствует или закончился.

Для пользователя логика должна быть продуктовой, а не технической.

---

## 7. UI-логика синхронизации

### 7.1. Главный принцип

Пользователю нужно показывать простые состояния.

Не нужно показывать внутреннюю техническую сложность синхронизации.

---

### 7.2. Состояния в UI

Возможные пользовательские состояния:

#### Локально

> Данные хранятся только на этом устройстве.

#### Синхронизация выключена

> Включите синхронизацию, чтобы пользоваться Kopim на разных устройствах.

Кнопка:

> Включить

#### Синхронизация включена

> Данные синхронизируются между устройствами.

#### Нужно действие

> Мы нашли данные на устройстве и в аккаунте. Выберите, что с ними сделать.

#### Ошибка синхронизации

> Не удалось синхронизировать данные. Ваши изменения сохранены на устройстве.

---

### 7.3. Pending changes

Pending changes не нужно показывать, если пользователь ничего не должен делать.

Правило:

* если всё работает штатно — не показываем;
* если изменения ждут сети — можно не тревожить пользователя;
* если идёт синхронизация — можно показать маленький индикатор;
* если есть риск потери или смешивания данных — показываем явное действие;
* если нужна авторизация, подписка или решение пользователя — показываем CTA.

---

### 7.4. Pull-to-refresh

Pull-to-refresh должен:

* отправлять локальные pending changes;
* подтягивать изменения из облака;
* показывать понятный результат.

Возможные сообщения:

* `Обновлено`;
* `Изменений нет`;
* `Не удалось синхронизировать`;
* `Нужно действие`.

---

## 8. Конфликты и защита данных

Текущая модель:

* в основном last-write-wins по `updatedAt`;
* есть registry/baseRemote проверки конфликтов;
* есть tombstone-семантика;
* для accounts используется особый `typeVersionAwareLww`.

Это хороший базовый подход.

Но простой LWW нельзя бездумно применять ко всем финансовым сущностям.

Особенно осторожно нужно относиться к:

* операциям;
* переводам;
* кредитным платежам;
* копилкам;
* связям между сущностями;
* сущностям, где удаление может порвать историю;
* сущностям, влияющим на баланс.

Главный принцип:

> Финансовое событие нельзя терять.

При опасной ситуации синхронизация должна блокироваться и объясняться пользователю.

UI-логика:

> Мы остановили синхронизацию, чтобы не смешать данные и не потерять операции. Выберите, что сделать дальше.

---

## 9. Онбординг

Текущий онбординг уже есть, сейчас его не трогаем.

В будущем можно улучшить, но не в рамках первого этапа подготовки local/cloud логики.

Планируемое улучшение:

* добавить готовый набор категорий;
* не заставлять пользователя вручную создавать базовые категории;
* дать возможность редактировать и удалять категории позже.

Базовый набор категорий расходов:

* Продукты;
* Кафе и рестораны;
* Транспорт;
* Дом;
* Связь и интернет;
* Здоровье;
* Одежда;
* Развлечения;
* Подписки;
* Подарки;
* Другое.

Базовый набор категорий доходов:

* Зарплата;
* Подработка;
* Подарки;
* Возврат долга;
* Проценты;
* Другое.

AI-онбординг в первой версии не нужен.

## 10. Главный экран

Главный экран должен быть дашбордом и финансовым помощником.

Пользователь за 5 секунд должен понять:

* сколько денег всего;
* какие категории близки к лимиту;
* что сейчас требует внимания;
* всё ли нормально с бюджетом.

---

### 10.1. Приоритет блоков главного экрана

1. Общий баланс.
2. Бюджет на месяц.
3. Проблемные категории.
4. Ближайшие регулярные платежи.
5. Последние операции.
6. Подсказки.

---

### 10.2. Бюджетный виджет

Бюджетный виджет должен показывать:

* общий лимит;
* проблемные категории;
* человеческие подсказки.

Пример:

> Бюджет на месяц
> Осталось 18 400 ₽ из 45 000 ₽
>
> Обрати внимание:
> Развлечения почти закончились — осталось 500 ₽.
> Продукты в норме — осталось 5 000 ₽.

Тон подсказок:

* человечный;
* спокойный;
* без давления;
* без обвинения пользователя.

Пример хорошего тона:

> Развлечения почти закончились — осталось 500 ₽.

Плохой тон:

> Вы слишком много тратите на развлечения.

---

## 11. Пустые состояния

Пустые состояния должны быть обучающими и сразу предлагать действие.

Пример:

> Добавьте первый счёт
> Так Kopim сможет показать общий баланс и помочь следить за расходами.

Кнопка:

> Добавить счёт

---

## 12. Текущая модель публичной Android-сборки

Модель:

* приложение устанавливается из Play Market как `storeProdLocalFirst`;
* по умолчанию работает локально;
* пользователь может пользоваться без регистрации;
* позже пользователь может включить облако;
* облако открывает синхронизацию, web, ИИ и расширенную аналитику;
* доступ управляется entitlement/subscription-логикой.

`offlineOnly` сохраняется как поддерживаемый автономный дистрибутив для privacy/offline сценариев, но больше не считается основным публичным Play flow.

| `offlineOnly` | Firebase/Auth/Firestore/Remote Config/cloud sync/AI transport отключены | `cloudSync`, `webApp`, `aiAssistant`, `advancedAnalytics` всегда `disabledByBuild` | `cloudUnavailableInBuild` | локальный режим, локальные CRUD, импорт/экспорт, UI "Локально" | cloud sign-in, sync, web access, AI, preflight continuation |
| `dev` | Firebase/Auth/Firestore/Remote Config/cloud sync/AI transport включены, dev Firebase | `requiresEntitlement`, `requiresSignIn`, `blockedByLocalData`, `enabled`, `unavailable` | `entitlementRequired`, `signedOut`, `blockedByLocalOnlyData`, `alreadyCloudEnabled`, `readyForNextStep`, `unknown` | demo/dev entitlement, cloud-capable auth, preflight + choice screen, execution сценарии (`enableCloudSync`, `startWithEmptyCloud`, `migrateLocalToCloud`) | merge local -> cloud (когда в облаке уже есть данные), server-backed entitlement, trial lifecycle |
| `storeProdLocalFirst` | Firebase/Auth/Firestore/Remote Config/cloud sync/AI transport включены, prod Firebase | `requiresEntitlement`, `requiresSignIn`, `blockedByLocalData`, `enabled`, `unavailable` | `entitlementRequired`, `signedOut`, `blockedByLocalOnlyData`, `alreadyCloudEnabled`, `readyForNextStep`, `unknown` | cloud-capable auth, preflight + choice screen, execution сценарии (`enableCloudSync`, `startWithEmptyCloud`, `migrateLocalToCloud`) | public subscription backend, merge local -> cloud (когда в облаке уже есть данные), paid lifecycle, web expired read-only |
| `web` cloud-capable build | технически те же cloud capabilities, что и у cloud-capable mobile build | `webApp` и `cloudSync` зависят от entitlement/gates; `isWebReadOnly` пока только projection | те же product states, что и у cloud-capable build, если экран preflight открыт | sign-in, cloud-capable routing, gate projection, capability model | настоящий write barrier для expired web entitlement, полноценный read-only режим, отдельная web matrix по операциям create/update/delete |

### Текущие capability-инварианты

* `AppCapabilities` отвечает только за технические возможности сборки.
* `FeatureAccess` отвечает за продуктовый доступ к sync/web/AI/advanced analytics.
* `DataMode` по-прежнему описывает текущее поведение данных, а не entitlement lifecycle.
* `offlineOnly` не должен запускать cloud/Firebase сценарии ни через UI, ни через pull-to-refresh.
* `dev` и `storeProdLocalFirst` остаются cloud-capable, но доступ к функциям идёт через gates, а не через raw flavor checks.

### Текущие preflight-state значения

Сейчас safe preflight flow умеет остановить пользователя и объяснить следующее безопасное действие, перенаправляя его на choice screen:

* `cloudUnavailableInBuild`
* `signedOut`
* `entitlementRequired`
* `blockedByLocalOnlyData`
* `readyForNextStep`
* `alreadyCloudEnabled`
* `unknown`

Дальше выбор сценария и запуск execution paths (`enableCloudSync`, `startWithEmptyCloud`, `migrateLocalToCloud`) вынесены в отдельные choice/execution слои. Экран preflight перенаправляет на choice screen, где пользователь может безопасно подтвердить желаемое действие (включая контролируемую миграцию локальных данных с резервным копированием и пошаговой выгрузкой, если в облаке нет других пользовательских данных).

### Что пока сознательно не реализовано

* полноценный merge local -> cloud (когда и на устройстве, и в облаке одновременно присутствуют пользовательские финансовые данные);
* сценарии `replaceLocalWithCloud` и `mergeLocalAndCloud` в execution flow (они пока заблокированы до реализации соответствующих алгоритмов слияния);
* server-backed entitlement и UID-backed claims lifecycle;
* trial lifecycle;
* настоящий web read-only barrier для expired entitlement;
* изменение `SyncContract`, Drift schema, outbox ordering, ownership guard и auth-sync workflow.
