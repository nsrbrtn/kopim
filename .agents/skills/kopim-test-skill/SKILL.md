# Kopim Test Skill

Этот навык регулирует разработку тестов, поддержание тестового покрытия и минимизацию токенов при их запуске.

## 1. Когда использовать этот навык
Активируйте этот навык при выполнении задач, которые затрагивают:
- Написание новых Unit, Widget или Integration тестов.
- Исправление упавших тестов.
- Валидацию изменений перед сдачей задачи (DoD).
- Изменение конфигураций тестирования.

## 2. Какие файлы читать в первую очередь
- Папка [test/](file:///home/artem/StudioProjects/kopim/test) — все тесты.
- Конфигурации моков и фейков (например, использование `fake_cloud_firestore` в `test/`).
- Раздел "Тестирование и качество" в `AGENTS.md`.

## 3. Запрещенные действия
- **Запрещено** коммитить изменения, если тесты в затронутом модуле падают.
- **Запрещено** бездумно запускать весь тестовый набор (`flutter test`) при локальных изолированных изменениях, так как это расходует ресурсы системы. Сначала запускайте целевые (targeted) тесты.
- **Запрещено** мокать внутреннее поведение классов Drift или Firestore вручную, если доступны готовые фейки (`fake_cloud_firestore` и in-memory базы Drift).
- **Запрещено** пропускать тесты через `@Skip()` без веской причины, согласованной с пользователем.

## 4. Какие проверки запускать
- **Целевой запуск тестов:**
  ```bash
  ./tool/ai_check.sh test/path_to_my_test.dart
  ```
- **Запуск всех тестов (только при изменениях в sync/db/auth/core или перед релизом):**
  ```bash
  ./tool/ai_check.sh --all
  ```

## 5. Типичные ошибки для предотвращения
- **Утечки времени (Time-travel bugs):** Тесты, завязанные на текущую дату и время (`DateTime.now()`), которые начинают падать в конце месяца/года или в определенные часы. Используйте инъекцию провайдеров времени или пакет `fake_async`.
- **Зависимость между тестами:** Когда запуск `test_A` влияет на результат `test_B` из-за общего глобального состояния или незакрытых стримов. Очищайте состояние в `tearDown` или `setUp`.
- **Нестабильные (flaky) Widget-тесты:** Забытый вызов `tester.pumpAndSettle()` при тестировании анимаций или переходов, из-за чего виджеты не успевают отрисоваться.

## 6. Ожидаемый результат (Expected Output)
- Новая бизнес-логика покрыта unit-тестами.
- Граничные UI-состояния (ошибки, пустые списки) покрыты widget-тестами.
- Сценарии интеграции (например, сохранение в локальную БД и фоновый синк) покрыты интеграционными тестами.
- Вывод `./tool/ai_check.sh` зеленый.
- Все тестовые файлы лежат в структуре, зеркальной `lib/` (например, тест для `lib/features/accounts/domain/use_cases/my_use_case.dart` должен быть в `test/features/accounts/domain/use_cases/my_use_case_test.dart`).
