# Kopim UI Skill

Этот навык используется для задач, которые затрагивают Flutter UI, экраны, виджеты, темы, дизайн-токены, адаптивную верстку, анимации, списки и presentation-слой на Riverpod.

## 1. Когда использовать этот навык

Используй этот навык, если задача затрагивает:

* создание или изменение экранов;
* создание или изменение переиспользуемых UI-компонентов;
* карточки, виджеты, bottom sheets, dialogs, forms;
* темы, цвета, типографику, отступы, радиусы, тени;
* дизайн-токены Figma;
* адаптивную верстку для Android, iOS, Web, Desktop;
* состояния UI через Riverpod;
* `ConsumerWidget`, `ConsumerStatefulWidget`, `ref.watch`, `ref.read`, `ref.listen`;
* анимации, скроллы, списки, `ListView`, `CustomScrollView`, `Sliver`;
* accessibility, читаемость, touch targets, empty/error/loading states.

Не используй этот навык для задач, которые меняют только Drift, sync, Firebase, release-конфиги или бизнес-логику без изменений UI.

## 2. Что читать в первую очередь

Не читай весь UI-слой. Начни с точечного поиска по названию экрана, виджета, feature, provider или текста интерфейса.

Обычно релевантны:

* `lib/core/theme/` — тема, цвета, типографика, дизайн-токены.
* `docs/components/` — документация по переиспользуемым UI-компонентам, если она есть.
* `tool/figma_theme/generate_tokens.dart` — генерация токенов из Figma, если задача затрагивает тему/токены.
* Файлы конкретной feature в `presentation`-слое.
* Релевантные providers/controllers/notifiers, которые используются конкретным экраном.
* Существующие похожие экраны и компоненты, чтобы сохранить стиль проекта.

Если путь к файлу изменился, найди актуальный путь через поиск по имени класса, компонента или feature.

## 3. Архитектурные правила UI

* UI не должен обращаться напрямую к Drift, Firebase, repositories или data sources.
* UI должен работать через presentation state/controllers/providers.
* Бизнес-логику не размещать внутри `build()`.
* Тяжёлые вычисления, форматирование, фильтрацию, сортировку и подготовку данных выносить из `build()` в controller/provider/model/helper.
* `ref.read` использовать для действий.
* `ref.watch` использовать для данных, от которых зависит отображение.
* `ref.listen` использовать для одноразовых эффектов: snackbar, navigation, dialog, error side effects.
* Сохранять существующий стиль feature и проекта.

## 4. Дизайн-система и темы

* Не прописывать ad-hoc цвета прямо в виджетах: `Colors.red`, `Colors.green`, `Color(0xFF...)`, если это не временный debug-кейс.
* Цвета, типографику и базовые стили брать из `Theme.of(context)`, `colorScheme`, `textTheme` или дизайн-токенов проекта.
* Не менять вручную сгенерированные токены, если они должны обновляться через генератор.
* Если меняются исходники токенов, запустить актуальную команду генерации токенов из документации проекта.
* Если команды нет в документации, проверить существующий способ запуска скрипта перед изменением workflow.
* UI должен корректно работать в светлой и тёмной теме, если обе темы поддерживаются проектом.
* Не ломать существующие цветовые роли и контрастность.

## 5. Адаптивность и состояние экранов

Для новых и изменяемых экранов проверь:

* small Android screen;
* large Android screen;
* Web/Desktop ширину, если экран доступен на Web/Desktop;
* keyboard/focus для форм;
* длинные тексты;
* большие суммы/числа;
* пустое состояние;
* loading state;
* error state;
* offline/disabled state, если экран связан с cloud/sync.

Не добавляй сложную адаптивность без необходимости. Сначала используй самое простое достаточное решение: `LayoutBuilder`, `ConstrainedBox`, `Wrap`, `Flexible`, `Expanded`, responsive breakpoints проекта, если они уже есть.

## 6. Производительность UI

* Используй `const` constructors, где это возможно и не ухудшает читаемость.
* Не создавай `NumberFormat`, `DateFormat`, регулярные выражения и тяжёлые объекты на каждый `build()` в списках.
* Для больших списков используй ленивые builders.
* `itemExtent` или `prototypeItem` используй только если размер элементов фиксированный или хорошо предсказуемый.
* Для больших state-объектов и часто обновляемых экранов используй `select()`, если виджету нужно только одно поле.
* Не дроби виджеты чрезмерно, если это ухудшает читаемость.
* Не оптимизируй преждевременно маленькие статичные экраны.

## 7. Локализация и тексты

* Пользовательские тексты должны идти через текущий механизм локализации проекта, если он используется в этой зоне.
* Не хардкодить повторяющиеся тексты в нескольких местах.
* Тексты ошибок, пустых состояний и подсказок должны быть понятными пользователю.
* Не менять тональность интерфейса без задачи.

## 8. Запрещённые действия

Запрещено без отдельной задачи или явного подтверждения:

* менять дизайн-систему глобально;
* менять базовые цвета/типографику приложения;
* переписывать существующие экраны “заодно”;
* менять business/domain/data слой ради UI без необходимости;
* добавлять новые зависимости для UI;
* менять navigation architecture;
* менять формат данных или sync-поведение ради отображения;
* вручную редактировать generated-файлы;
* добавлять тяжёлые анимации без проверки производительности.

## 9. Проверки

После обычных UI-изменений:

```bash
./tool/ai_check.sh
```

Если менялись generated providers, `riverpod_annotation`, `freezed`, `json_serializable`, `.g.dart`, `.freezed.dart` или кодогенерируемые модели:

```bash
dart run build_runner build --delete-conflicting-outputs
./tool/ai_check.sh
```

Если менялись дизайн-токены:

```bash
# Используй актуальную команду генерации токенов проекта.
# Если она не задокументирована, сначала найди способ запуска в tool/figma_theme/ или README.
./tool/ai_check.sh
```

Если добавлен или изменён сложный переиспользуемый компонент, добавь или обнови релевантные widget/golden tests, если такая практика уже есть в проекте.

## 10. Expected Output

После выполнения UI-задачи агент должен вывести:

* какие экраны/виджеты изменены;
* какие providers/controllers затронуты;
* менялась ли тема или дизайн-токены;
* поддержаны ли светлая/тёмная темы;
* как обработаны loading/error/empty states, если они релевантны;
* как учтена адаптивность;
* какие проверки запущены;
* результат проверок;
* какие риски или ручные follow-up остались.

Если добавлен новый переиспользуемый UI-компонент, обнови `docs/components/`, если такая документация уже ведётся для подобных компонентов.
