# ExecPlan: Типизированные счета целей с фокусом на копилки

## Context and Orientation
- Цель: реализовать первый инкремент идеи «отдельный тип счета под каждую цель», начиная с копилок: гарантировать отдельный счет для каждой копилки и добавить аналитический виджет тренда накоплений.
- Область кода: `lib/features/savings/**`, `lib/features/analytics/**`, `lib/core/data/database.dart`, `lib/core/services/*sync*`, `docs/logic/**`, `docs/components/**`.
- Контекст/ограничения:
  - Для каждой копилки должен быть отдельный счет (1:1).
  - Пополнение копилки пока только переводом.
  - Баланс копилки считается по транзакциям.
  - Аналитика накоплений: прирост по дням/неделям/месяцам, по умолчанию 30 дней, агрегировано по всем копилкам.
  - Создание копилки автоматически создает счет.
  - Архивирование сохраняется как текущий механизм.
  - Синхронизация с Firebase обязательна.
- Риски:
  - Двойной учет переводов в аналитике (источник/получатель).
  - Регрессии в балансе счетов при откатах/удалении транзакций.
  - Несогласованность `saving_goals.account_id` и фактических `accounts`.

## Interfaces and Dependencies
- Внешние сервисы (Firebase/Analytics/Crashlytics):
  - Firestore sync через `SyncService`/`AuthSyncService`.
  - События `AnalyticsService` по копилкам.
- Локальные зависимости (Drift, codegen):
  - Drift таблицы `saving_goals`, `goal_contributions`, `transactions`, `accounts`.
  - Riverpod providers в `analytics_providers.dart`.
  - L10n (`app_ru.arb`, `app_en.arb`) при добавлении новых текстов.
- Затрагиваемые API/модули:
  - `SavingGoalRepositoryImpl` (создание/поддержание счета копилки).
  - `AnalyticsScreen` вкладка накоплений.
  - Потоки `watchSavingGoalsUseCase` и `watchTransactions`.

## Validation and Acceptance
- Команды проверки:
```bash
dart format --set-exit-if-changed .
flutter analyze
flutter test --reporter expanded test/features/savings/data/saving_goal_repository_impl_test.dart
flutter test --reporter expanded test/features/analytics/presentation/controllers/analytics_providers_test.dart
flutter test --reporter expanded test/features/analytics/presentation/analytics_screen_test.dart
```
- Acceptance criteria:
  - Вкладка «Накопления» на экране аналитики показывает реальный тренд, а не заглушку.
  - Тренд переключается по day/week/month и по умолчанию показывает 30 дней.
  - Сводные метрики накоплений считаются агрегировано по всем активным копилкам.
  - У каждой копилки есть отдельный счет типа `savings`, создание автоматическое.
  - Sync/outbox сценарии не ломаются.

## Idempotence and Recovery
- Что можно безопасно перезапускать:
  - `dart format`, `flutter analyze`, targeted `flutter test`.
- Как откатиться/восстановиться:
  - откатить изменения в `features/analytics/**` и `features/savings/**`;
  - при проблемах в UI временно вернуть заглушку вкладки накоплений.
- План rollback (для миграций):
  - для этого инкремента новая миграция не обязательна;
  - если потребуется изменение схемы в следующем этапе, добавить отдельный rollback-план на уровне Drift.
