# ExecPlan: Реализация функции «Финансовый индекс» на экране обзора

## Context and Orientation
- Цель: реализовать рабочую функцию расчёта и отображения `Финансового индекса (0–100)` на экране `OverviewScreen`.
- Область кода:
  - `lib/features/overview/presentation/overview_screen.dart`
  - `lib/features/home/` (источники данных: сводка/транзакции)
  - `lib/features/analytics/` (агрегации по месяцам, если переиспользуем)
  - `lib/l10n/*.arb`
  - `test/features/overview/` и `test/features/overview/domain/`
  - `docs/logic/` и `docs/components/`
- Контекст/ограничения:
  - сейчас `OverviewScreen` — статичный UI-макет без функционала;
  - нужно сохранить текущую визуальную структуру карточки индекса;
  - под числом индекса требуется столбчатый график по месяцам;
  - под графиком требуется чип с текстом «сколько баллов в этом месяце»;
  - цвета и шрифты только из темы.
- Риски:
  - неполные данные у новых пользователей;
  - нестабильный индекс из-за шумных транзакций;
  - риск тяжёлых вычислений в UI-потоке.

## Interfaces and Dependencies
- Внешние сервисы: нет.
- Локальные зависимости:
  - Riverpod (`Provider/StreamProvider`),
  - Drift/репозитории транзакций/счетов/бюджетов,
  - существующие модели денег (`MoneyAmount`).
- Затрагиваемые API/модули:
  - `OverviewScreen.routeName` (без изменения маршрута),
  - новые use-case/провайдеры для индекса,
  - локализация новых подписей и состояний.

## Предложение по источникам данных факторов
1) Контроль бюджета (`budgetScore`, вес 30%)
- Источник:
  - активные бюджеты текущего месяца из модуля `features/budgets` (репозиторий/провайдеры бюджета),
  - операции расходов за месяц из `transactionRepositoryProvider`.
- Нужные метрики:
  - `plannedExpense` — суммарный лимит бюджетов за месяц;
  - `actualExpense` — фактические расходы по бюджетным категориям;
  - `overspendRatio = max(0, actualExpense - plannedExpense) / max(plannedExpense, 1)`.
- Предлагаемый расчёт:
  - `budgetExecution = clamp(100 * (1 - actualExpense / max(plannedExpense, 1)), 0, 100)`;
  - `budgetScore = clamp(budgetExecution - overspendRatio*40, 0, 100)`.
- Fallback:
  - если бюджетов нет, использовать нейтральный `budgetScore = 55` и пометить низкую уверенность.

2) Подушка безопасности (`safetyScore`, вес 30%)
- Источник:
  - ликвидный остаток по счетам из `accounts` (исключая liability-счета: `credit`, `credit_card`, `debt`);
  - средние месячные расходы за последние 3 месяца из транзакций;
  - цель резерва из настроек overview (если будет добавлена) либо дефолт `targetMonths = 6`.
- Нужные метрики:
  - `liquidAssets`;
  - `avgMonthlyExpense3m`;
  - `monthsCovered = liquidAssets / max(avgMonthlyExpense3m, 1)`;
  - `reserveProgress = monthsCovered / targetMonths`.
- Предлагаемый расчёт:
  - `coverageScore = clamp((monthsCovered / targetMonths) * 100, 0, 100)`;
  - `goalScore = clamp(reserveProgress * 100, 0, 100)`;
  - `safetyScore = round(coverageScore*0.7 + goalScore*0.3)`.
- Fallback:
  - если расходов почти нет, ограничивать `monthsCovered` верхним потолком (например 12) и не давать сразу 100.

3) Динамика месяца (`dynamicsScore`, вес 20%)
- Источник:
  - помесячные агрегаты расходов из `analytics`/`transactionRepositoryProvider`;
  - текущий и предыдущий месяц;
  - тренд за N последних месяцев (рекомендуемо N=6).
- Нужные метрики:
  - `currentExpense`, `previousExpense`;
  - `deltaRatio = (previousExpense - currentExpense) / max(previousExpense, 1)` (положительное = улучшение);
  - `trendSlope` по ряду расходов (линейный тренд или упрощённое сравнение 3 последних месяцев).
- Предлагаемый расчёт:
  - `deltaScore = clamp(50 + deltaRatio*100, 0, 100)`;
  - `trendScore = clamp(50 + (-trendSlopeNormalized)*50, 0, 100)`;
  - `dynamicsScore = round(deltaScore*0.6 + trendScore*0.4)`.
- Fallback:
  - если нет предыдущего месяца, брать `dynamicsScore = 50` (нейтрально).

4) Поведенческая дисциплина (`disciplineScore`, вес 20%)
- Источник:
  - ежедневная активность из `transactions` за последние 30 дней;
  - streak из последовательности дней с минимум одной записью.
- Нужные метрики:
  - `activeDays30`;
  - `consistencyRatio = activeDays30 / 30`;
  - `streakDays` (дни подряд до текущей даты).
- Предлагаемый расчёт:
  - `consistencyScore = clamp(consistencyRatio*100, 0, 100)`;
  - `streakScore = clamp((streakDays / 14) * 100, 0, 100)` (14 дней = максимум по streak в MVP);
  - `disciplineScore = round(consistencyScore*0.6 + streakScore*0.4)`.
- Анти-накрутка:
  - считать активным только факт наличия дня, а не количество операций в дне.

5) Источник для месячного столбчатого графика индекса (под текущий UI)
- Источник:
  - вычислять `FinancialIndexMonthlyPoint` по каждому месяцу окна (например, последние 6 или 12 месяцев),
  - для каждого месяца запускать тот же агрегатор факторов на срезе этого месяца.
- Выход:
  - массив: `[{month, score}]`, где `score` в `0..100`;
  - текущий месяц выделяется в UI как активный столбец;
  - чип под графиком берёт значение `currentMonthScore`.

## Что нужно реализовать в приложении сейчас (минимальный рабочий scope)
1) Доменная модель индекса:
- итоговый индекс `totalScore (0..100)`;
- баллы 4 факторов: `budget/safety/dynamics/discipline`;
- статус (`risk/unstable/stable/growth`) + microcopy;
- массив месячных точек для столбчатого графика (например, 6–12 месяцев).

2) Логику расчёта факторов (MVP):
- Контроль бюджета (30%): выполнение бюджета + штраф за перерасход.
- Подушка безопасности (30%): месяцы покрытия расходов + прогресс к цели резерва.
- Динамика месяца (20%): лучше/хуже предыдущего месяца + тренд трат.
- Дисциплина (20%): streak дней + регулярность записей.

3) Агрегации и провайдеры:
- провайдер текущего индекса;
- провайдер истории индекса по месяцам для столбчатого графика;
- нормализация и clamp в `0..100`.

4) Подключение к текущему UI карточки:
- заменить захардкоженное `74/100` на данные провайдера;
- заменить декоративные столбцы на реальные месячные значения индекса;
- добавить под графиком чип: `"{N} баллов в этом месяце"`;
- реализовать цветовую зону карточки по диапазонам и microcopy состояния.

5) Локализация и тесты:
- ключи для microcopy, чипа месяца, fallback-состояний;
- unit-тест формулы и порогов;
- widget-тест отображения графика/чипа/статуса.

## Concrete Steps
1) Зафиксировать формулу и границы в коде:
- `index = budget*0.30 + safety*0.30 + dynamics*0.20 + discipline*0.20`.
- Диапазоны статусов:
  - `0–40` → «Финансовый риск»
  - `40–60` → «Нестабильно»
  - `60–80` → «Стабильно»
  - `80–100` → «Уверенный рост»

2) Создать доменные сущности/DTO:
- `FinancialIndexSnapshot` (итог + breakdown + статус);
- `FinancialIndexMonthlyPoint` (месяц + score);
- `FinancialIndexSeries` (история + текущий месяц).

3) Реализовать use case расчёта:
- собрать входные данные из репозиториев;
- рассчитать 4 подскора;
- собрать итоговый score и историю по месяцам;
- обработать случай недостатка данных (fallback + нейтральный статус).

4) Создать Riverpod-провайдеры:
- `financialIndexProvider`;
- `financialIndexMonthlySeriesProvider`.

5) Обновить текущий UI `_FinancialIndexCard`:
- крупное число из `totalScore`;
- под числом — `Bar`-график по месяцам;
- под графиком — чип `"{N} баллов в этом месяце"`;
- статус справа от числа (из microcopy);
- цвета столбцов и чипа по зоне индекса.

6) Добавить l10n-ключи:
- статусы индекса;
- шаблон чипа месяца;
- заглушки для пустых данных.

7) Добавить тесты:
- unit: формула/границы/кламп/пример (80,60,70,90 => 74);
- widget: рендер столбцов по месяцам и корректного чипа;
- edge: отсутствие данных, нулевые расходы, резкие всплески.

8) Документация:
- `docs/logic/` — методика расчёта;
- `docs/components/` — поведение карточки индекса.

## Возможные сложности
- Нехватка данных для честного расчета подушки/дисциплины у новых пользователей.
- Разная частота ведения учета приводит к скачкам индекса.
- Категории/переводы/кредитные операции могут искажать фактор «Динамика».
- Производительность при пересчете истории по месяцам на больших объёмах транзакций.
