# ExecPlan: Аудит бизнес-логики приложения

## Context and Orientation
- Цель: провести сквозной аудит бизнес-логики по feature-модулям, выявить ошибки, логические расхождения, риски регрессий, слабые места синхронизации и неиспользуемые части кода, а затем подготовить итоговый отчет.
- Область кода: `lib/features/**`, `lib/core/**`, связанные `docs/logic/**`, `test/**`.
- Контекст/ограничения: аудит должен опираться на фактический код, тесты, инварианты и архитектурные документы; автоген и секреты не изменять; итогом является отчет в файле.
- Риски: большой объем кода, частично устаревшая документация, наличие модулей с неполным тестовым покрытием, ложноположительные выводы по "неиспользуемому" коду без проверки ссылок.

## Interfaces and Dependencies
- Внешние сервисы (Firebase/Analytics/Crashlytics): Firebase Auth, Firestore, локальные sync-адаптеры.
- Локальные зависимости (Drift, codegen): Drift, Riverpod generator, Freezed, build_runner.
- Затрагиваемые API/модули: feature-first модули приложения, локальная БД, sync-слой, доменные use case, контроллеры Riverpod.

## Plan of Work
- Зафиксировать структуру аудита и обязательные критерии проверки.
- Собрать карту модулей, зависимостей, тестового покрытия и потенциального dead code.
- Проверить критичную бизнес-логику по каждому модулю и сопоставить с инвариантами.
- Подготовить письменный отчет с приоритизацией проблем и рекомендациями.

## Concrete Steps
1) Создать ExecPlan и определить чек-лист аудита: доменная логика, синхронизация, инварианты, тесты, мертвый код.
2) Собрать список feature-модулей, ключевых use case, репозиториев и связанных тестов.
3) Пройти по модулям и зафиксировать логические дефекты, риски и пробелы покрытия.
4) Проверить наличие потенциально неиспользуемых модулей, экранов, провайдеров, сервисов и документов.
5) Сформировать итоговый отчет в `docs/logic/` с фактами, примерами и приоритетами.

## Validation and Acceptance
- Команды проверки:
```bash
dart format --set-exit-if-changed .
flutter analyze
flutter test --reporter expanded
```
- Acceptance criteria:
  - Есть отдельный отчет с перечислением найденных проблем, рисков и неиспользуемых частей.
  - Для каждого вывода указаны конкретные файлы/модули и причина.
  - Выводы разделены по степени критичности и снабжены рекомендациями.

## Idempotence and Recovery
- Что можно безопасно перезапускать: сбор структуры файлов, поиск по репозиторию, `flutter analyze`, тесты, обновление ExecPlan, подготовку отчета.
- Как откатиться/восстановиться: удалить созданные документы аудита.
- План rollback (для миграций): не требуется, схема БД не меняется.

## Surprises & Discoveries
- `flutter analyze` не показал ошибок, но полный `flutter test` падает с большим числом регрессий.
- В `credits` найден незавершенный `UpdateCreditPaymentUseCase`, при этом UI уже позволяет редактировать составные транзакции платежной группы.
- В `savings` обнаружена ошибка масштаба суммы: `appliedDelta / 100` ломает валюты с scale != 2.
- В `transactions` переводы между счетами не валидируют валютный контекст.
- Найден слой dead/legacy code: дубли `watch_accounts_use_case`, неиспользуемые виджеты/экраны и отсоединенный use case для overview goal focus.
